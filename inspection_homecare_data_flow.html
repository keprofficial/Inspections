Inspection App → Homecare Data Flow

This map shows how flat, individual, and society inspections connect to Supabase, where critical issues are pushed, and how Homecare can read them later.

1. Inspection Types And Checklist Mapping

Flat Inspection Code: INF######

Select society → block → flat. Checklist kind is flat. Stored against the selected flat property ID.

Individual Home Inspection Code: INP######

Enter property owner details manually. Checklist kind is still flat, so DB checklist rows are not duplicated.

Society Inspection Code: INS######

Select society only. Checklist kind is society, generated from Sheet2: Apartment Inspection Checklist.

DB mapping table inspection_type code_prefix checklist_kind Why
inspection_app_types flat INF flat Normal apartment/flat inspection.
inspection_app_types individual INP flat Different reporting code, same room-wise checklist.
inspection_app_types society INS society Common areas need a separate checklist.

2. End-To-End Submit Flow

1

Login

Inspector signs in using mobile + password through inspection_app_login.

2

Select Scope

App sets inspection_type, inspection_code, and selected property_id.

3

Start Inspection

inspection_app_start inserts into inspections.

4

Capture Evidence

Photos upload to Supabase Storage and public URLs are saved in checklist state.

5

Final Submit

PDF uploads, critical issues insert into inspection_issues, and Homecare reads those rows.

3. Core Tables And What They Store

Table / Storage Written By Main Data Stored Used By Homecare
properties Existing property DB Society, block, flat IDs and codes. Yes. Homecare joins critical issues to property context.

storage: inspection-photos Inspection app upload Images and PDF files. Homecare uses public URLs stored in DB rows.

4. Critical Issue Push Logic

On final submit, the app does not push every checklist item to Homecare. It pushes only completed items where severity is critical. Each selected service creates one row in inspection_issues.
App field DB column Example Homecare purpose
Inspection ID inspection_issues.inspection_id UUID from inspections.id Join to inspection, property, report PDF.
Area + item + notes description Kitchen - Gas leak check - technician notes Display issue context to Homecare/resident.
Selected service service_code, linked_service_id KS274 Create recommended Homecare service action.
Estimated cost estimated_cost 1500 Stored for DB/estimation, not shown in PDF.
Material codes material_codes {MAT001, MAT014} Later material planning or technician app use.
Uploaded photos photo_urls Array of public image URLs Homecare displays proof photos per critical issue.
Generated issue ref issue_ref stable hash from inspection + area + item + service De-duplicate/update same issue-service row.

5. Flat Critical Issues vs Society Critical Issues

Flat critical issue push

inspections.property_id points to a flat row in properties. Homecare fetches critical issues by that flat property ID or by recent critical issues.

inspection_type: flat code: INF###### checklist: flat
Society critical issue push

inspections.property_id points to a society row in properties. The same inspection_issues table stores the society critical issues, so Homecare can show society-level repairs and services.

inspection_type: society code: INS###### checklist: society 
-- Homecare / Resident App: fetch critical issues for one flat or society property
select
  i.id as inspection_id,
  coalesce(i.inspection_code, i.inspection_ref) as inspection_code,
  i.inspection_type,
  i.full_report_pdf_url,
  p.id as property_id,
  p.name as property_name,
  p.type as property_type,
  ii.issue_ref,
  ii.description,
  ii.service_code,
  s.name as service_name,
  ii.photo_urls,
  ii.estimated_cost,
  ii.material_codes,
  ii.status
from public.inspection_issues ii
join public.inspections i on i.id = ii.inspection_id
join public.properties p on p.id = i.property_id
left join public.services s on s.service_code = ii.service_code
where lower(ii.severity) = 'critical'
  and i.property_id = :selected_flat_or_society_property_id
order by i.conducted_at desc, ii.issue_ref;
